目标与交付

以“可稳定卖货 + AI 能持续优化”为第一优先级：站点能完成真实交易闭环，且关键转化链路可观测、可迭代。

  • 交易主链跑通：/shop、/product、/cart、/checkout、/order 全链路真实数据可用。
  • 内容入口转化化：guides / app-control / quiz 形成明确导购路径。
  • AI 控制平面接入“结果型信号”：不仅看行为（CTR），也能看下单、弃单等更贴近业务结果的指标。
  • Ops 总控视图可用：能直接看到“待决 / 待审 / 执行中 / 观察中 / 风险未收敛”。

关键假设

  • Sanity 和 Medusa 的生产配置（项目 ID、token、支付、库存）由外部环境提供；计划中只负责接入与验证。
  • 第一阶段不追求评价体系、搜索、多语言、会员等扩展功能。
  • AI 自动执行维持“人授权”的边界：自动起草、自动准备、自动追踪；合并/发布/回滚仍需明确权限节点。
风险提示

当前 AI 治理能力进展领先于电商主链成熟度。计划优先把交易主链打稳，否则 AI 的优化会落在半成品链路上。

实施分期（建议 4 周节奏）

阶段 目标 主要交付
第 0 周
基线稳定
 把本地/联调环境跑稳，避免后续被环境噪音打断。
  • 统一启动：npm run dev:all
  • web build 稳定通过（依赖完整）
  • Medusa 本地 seed 可复现
第 1 周
交易主链
 去掉 mock fallback 的关键路径，确保商品/价格/购物车/订单的真实读取与写入。
  • /shop、/product 使用真实 Medusa 数据
  • cart / checkout 基本可用（至少 sandbox）
  • 订单查询页能展示真实订单状态
第 2 周
转化入口
 把内容与 AI 入口变成导购入口：quiz 输出可落到产品/套装，guides 与专题页能引导到购买链路。
  • quiz → 推荐商品/collection/bundle 的固定路由
  • guides → 商品卡片/CTA 联动
  • app-control / bundles 页面与真实商品绑定
第 3 周
AI 接结果
 把 signals 从“行为信号”扩到“结果信号”，让控制平面能对转化与弃单提出建议并形成治理对象。
  • 新增订单/支付/弃单类 signals
  • 至少 1 条非 AI concierge 的第二条样板线接入治理壳
  • monitoring 用 group 统一展示多线治理状态

说明：4 周是建议节奏。若资源有限，可以把第 2、3 周合并，但不要把“交易主链”往后挪。

工作流拆分

交易主链

  • apps/web/src/lib/commerce/*：真实 Medusa 数据读写（products/cart/checkout/orders）。
  • apps/web/src/app/(shop)/*：页面与错误态/空态打磨。
  • apps/medusa/apps/backend：seed/migrate/本地联调稳定性。

内容入口转化

  • apps/web/src/app/(marketing)/*：专题/指南导购路径。
  • apps/web/src/app/(shop)/quiz/*：quiz 输出与商品/套装路由。
  • apps/web/src/lib/seo/*：SEO 与导购组件共存。

AI 与运营（全站口径）

信号

  • 从 view/cta/add_to_cart 扩到 checkout_start、payment_success、checkout_dropoff 等。
  • 让 signals 能覆盖“内容 → 商品 → 购物车 → 订单”的路径归因。

治理与平台化

  • apps/control-plane/src/signals/governance-patterns/*：治理壳（观测窗口/复判）。
  • apps/control-plane/src/ops/monitoring.js：总控台 groups 结构。
  • apps/web/src/app/(ops)/ops/monitoring：统一渲染各条治理线。

完成标准

  • 从首页/内容入口进入，用户能完成一次从商品浏览到下单的完整链路。
  • 关键路径埋点可复现：能在 monitoring 中看到转化链路的主要信号。
  • Ops 能在一个页面（monitoring + proposal）完成“发现 → 建议 → 执行 → 观察 → 结论”的闭环阅读。
  • 至少两条业务线使用同一套治理骨架（AI concierge + 第二条线）。
